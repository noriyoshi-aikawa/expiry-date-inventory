# 要件定義書：賞味期限・ロット在庫管理機能

## 1. プロジェクトの概要

### 1-1. 目的

食品EC事業者が現場で抱える「賞味期限・ロット管理の属人化」を解消し、EC上で賞味期限・ロット単位の在庫管理を完結させることを目的とする。
FEFO（期限近い順の先入先出）自動引当・残日数バリデーション・アラートメールを提供することで、誤出荷クレームの防止と廃棄ロス削減を実現し、食品EC加盟店のGMV向上に貢献する。

### 1-2. 背景

| 課題 | 備考 |
|---|---|
| makeshopの在庫管理は「数量」のみで、賞味期限・ロットの概念が存在しない | 食品ECに不可欠な「時間軸」管理が既存機能では代替不可能 |
| 出荷可否判断が担当者の目視・記憶に完全依存している | 担当者属人化による退職リスク・引き継ぎ困難が発生 |
| FEFO（期限近い順の先入先出）が実現できず、残日数基準以下の誤出荷を防ぐ仕組みがない | 誤出荷によるクレーム・返品・信頼損失につながっている |
| 廃棄予定在庫の訳あり品化・割引販売への切り替えが手動対応のみ | 売上化できる在庫を廃棄しており、GMVの機会損失が発生している |
| 食品EC市場は4兆円規模で成長中。Shopifyの食品商習慣非対応に課題を感じる事業者が国産カートへ回帰するニーズが存在する | ヒアリング5社中3社（オーム乳業・エスポア・王子サーモン）が賞味期限管理を最優先・高優先と評価 |

## 2. 機能要件

### 2-1. 改修対象画面一覧

| # | 画面名 | 区分 | 改修内容の概要 |
|---|---|---|---|
| 1 | ロット管理画面 | 管理画面（新規追加） | 商品バリエーション単位でロット情報（賞味期限・製造日・入荷日・ロット在庫数）を登録・一覧表示する画面 |
| 2 | 商品情報一括アップロードCSV | 管理画面（既存改修） | ロット情報（賞味期限・製造日・入荷日・ロット在庫数）の一括登録に対応するためのCSVフォーマット拡張 |
| 3 | 商品設定画面（バリエーション設定） | 管理画面（既存改修） | 商品ごとの「出荷可能最低残日数」設定項目を追加 |
| 4 | 在庫アラートメール設定 | 管理画面（既存改修） | 既存の在庫アラート機能を拡張し、賞味期限残日数アラートの閾値設定を追加 |
| 5 | 注文処理（引当ロジック） | 管理画面（内部処理改修） | 注文受付時のFEFO自動引当ロジックを実装 |

### 2-2. 各機能の詳細

#### 機能①：ロット登録

商品バリエーション単位にロット情報を登録する機能。手動入力とCSV一括登録の両方に対応する。

| 項目名 | 設定内容 | 初期値 |
|---|---|---|
| ロットNo. | ロットを識別するための任意の番号・コード（英数字） | ー（必須入力） |
| 賞味期限 | ロットの賞味期限（日付形式：YYYY-MM-DD） | ー（必須入力） |
| 製造日 | ロットの製造日（日付形式：YYYY-MM-DD） | ー（任意） |
| 入荷日 | ロットの入荷日（日付形式：YYYY-MM-DD） | ー（任意） |
| ロット在庫数 | 当該ロットの在庫数量（整数） | ー（必須入力） |
| 紐付け商品バリエーション | 登録対象の商品バリエーション（SKU） | ー（必須選択） |

> [!NOTE]
> 「ロット」とは同じ条件で製造・入荷された商品のひとまとまりを指す。同じ商品でも製造日が異なれば賞味期限が異なるため、ロット単位での在庫管理によりFEFO制御が初めて可能となる。
> CSVフォーマットの詳細仕様は着手前にオーム乳業へ追加ヒアリングを実施したうえで確定する（要確認）。

#### 機能②：FEFO自動引当

注文受付時に、賞味期限が近い順のロットから自動的に引当を行うロジック。

| 項目名 | 設定内容 |
|---|---|
| 引当方式 | FEFO（First Expired, First Out）：賞味期限が近いロットを優先して引当 |
| 複数ロットまたぎ対応 | 1注文が複数ロットにまたがる場合もトランザクション管理で在庫整合性を担保 |
| 同時注文時のRace Condition対策 | 排他制御（行ロック等）を実装し二重引当を防止（技術詳細はTech Leadと設計） |

#### 機能③：残日数バリデーション

商品ごとに出荷可能な最低残日数を設定し、基準を下回るロットを引当対象から自動除外する機能。

| 項目名 | 設定内容 | 初期値 |
|---|---|---|
| 出荷可能最低残日数 | 注文受付時に引当対象とするロットの最低残日数（単位：日）。この日数を下回るロットは引当から除外される | ー（任意設定。未設定時は残日数制限なし） |
| バリデーション実行タイミング | 注文受付時（FEFO引当処理と同時） | ー |
| 除外ロットの扱い | 引当対象から除外するのみ。在庫数はそのまま保持（廃棄・訳あり品化はPhase2対応） | ー |

> [!NOTE]
> 残日数の計算基準（出荷日・受注日・配送予定日のいずれを基準とするか）は着手前にオーム乳業へ追加ヒアリングを実施したうえで確定する（要確認）。

#### 機能④：アラートメール（賞味期限切迫通知）

設定した残日数閾値を下回ったロットが発生した際に、管理者へ自動メール通知する機能。既存の在庫アラートメール機能を拡張して実装する。

| 項目名 | 設定内容 | 初期値 |
|---|---|---|
| アラート閾値（残日数） | アラートメールを送信する残日数の基準（単位：日） | ー（ショップ様が任意設定） |
| 送信タイミング | 日次バッチ処理（時刻は要確認） | ー |
| 送信先 | ショップ様の管理者メールアドレス | 既存の在庫アラートメール設定に準拠 |
| メール内容 | 閾値を下回ったロットの商品名・バリエーション・ロットNo.・賞味期限・在庫数を一覧形式で通知 | ー |

### 2-3. 画面遷移フロー

#### ロット登録フロー（手動）

```
管理画面ログイン
    └→ 商品管理
        └→ 商品一覧
            └→ 対象商品を選択
                └→ バリエーション設定
                    └→ ロット管理タブ（新規追加）
                        └→ ロット新規登録ボタン
                            └→ ロット情報入力フォーム
                                ├→ 入力完了 → 保存 → ロット一覧に反映
                                └→ バリデーションエラー → エラーメッセージ表示 → 再入力
```

#### 注文受付時の引当フロー（自動処理）

```
ユーザー様がショップにて注文確定
    └→ 注文受付処理
        └→ 対象商品にロット管理設定があるか判定
            ├→ ある場合：FEFO引当ロジック実行
            │       └→ 残日数バリデーション通過ロットを賞味期限昇順でソート
            │           ├→ 在庫充足 → 引当確定 → 注文完了
            │           └→ 在庫不足 → 在庫不足エラー（既存フローに準拠）
            └→ ない場合：既存の通常引当フローへ
```

#### アラートメール送信フロー（自動処理）

```
日次バッチ実行
    └→ 全ロットの残日数を計算
        └→ アラート閾値以下のロットを抽出
            ├→ 対象ロットあり → 管理者へアラートメール送信
            └→ 対象ロットなし → 処理終了
```

### 2-4. やらないこと（今回スコープ外）

| 項目 | 理由 |
|---|---|
| 訳あり品自動切替・価格変更 | Phase 2（+2ヶ月）として後続スプリントで対応 |
| 管理者承認オプション（訳あり品切替時） | Phase 2対応 |
| EC上の切迫シグナル表示（ショップ画面） | Phase 2対応 |
| 基幹システムとのAPI定時連動 | Phase 3（+3〜4ヶ月）。オーム乳業の2026年8月基幹刷新に合わせて対応 |
| ロット別出荷レポート | Phase 3対応 |
| 複数倉庫・ロケーション管理 | 現行makeshopの倉庫管理仕様の範囲外 |
| 温度帯管理との統合 | 別途温度帯担当チームとのDBスキーマ合同設計を経て検討 |

## 3. 非機能要件

### 3-1. 操作性・UX要件

- ロット登録画面はショップ様が専門知識なく直感的に入力できるシンプルなUIとする
- CSVアップロードはエラー時に「どの行・どの項目が誤っているか」を明示するバリデーションメッセージを表示する

### 3-2. パフォーマンス要件

- FEFO引当処理は注文受付の既存レスポンスタイムを著しく劣化させないこと（詳細はTech Leadと数値定義・要確認）
- 日次バッチ処理はショップ様の注文ピーク時間帯を避けた時間に実行すること（要確認）

### 3-3. セキュリティ・コンプライアンス要件

- 同時注文時のRace Conditionを排他制御で防止し、二重引当・在庫マイナスが発生しないこと
- ロットデータは注文データと同等の取り扱いとし、アクセス権限は管理者・副管理者に限定する

## 5. データ要件

### 5-1. データ保持期限・管理方針

| データ種別 | 保持方針 | 削除タイミング |
|---|---|---|
| ロット情報（賞味期限・製造日・入荷日・在庫数） | 永続保持 | ショップ様が手動削除するまで保持（在庫ゼロになっても履歴として残す） |
| FEFO引当履歴（どの注文でどのロットを引当したか） | 永続保持 | 注文データの保持ポリシーに準拠（要確認） |
| 日次バッチ実行ログ | （要確認） | （要確認） |

### 5-2. DBスキーマ設計方針

- 新規ロットテーブルを追加し、既存の商品バリエーション（SKU）テーブルと外部キーで紐付ける
- 全加盟店の在庫データに影響するDBマイグレーションが必要なため、着手前に温度帯担当チームと合同スキーマ設計会議を実施すること（必須）

> [!WARNING]
> DBスキーマ変更は全加盟店の在庫データへのマイグレーションを伴う。移行計画・ロールバック手順を事前に策定すること。

## 6. 業務要件

| 設定内容 | 反映タイミング | 備考 |
