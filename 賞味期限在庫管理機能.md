# 賞味期限・ロット在庫管理機能 企画書

作成日：2026年6月1日  
作成者：AIプロダクトチーム

---

## 1. 機能名

賞味期限・ロット在庫管理機能（食品BtoB業務パック 中核機能）

---

## 2. 背景 / 課題

- **現在の在庫管理は「数量」のみ：** makeshopの在庫管理は数量だけで管理されており、食品ECに不可欠な「時間軸（賞味期限・ロット）」の概念が存在しない。
- **現場は目視・記憶・手作業：** 生クリーム（賞味期限20日）を扱う食品ECでは「出荷最低残8日」ルールをEC上で強制できず、出荷前日に倉庫担当が実在庫を目視でロット当て直している。担当者が変われば即崩壊するリスクがある。
- **食品EC市場は4兆円規模で成長中：** BtoB・製造直販・D2C食品事業者はShopifyの食品商習慣非対応に課題を感じており、国産カートへの回帰ニーズが存在する。差別化の核心になりうる。
- **「食品BtoB業務パック」の中核機能：** ヒアリング5社中3社が賞味期限管理を最優先・高優先と評価している。

---

## 3. 解決したい問題

| 項目 | 内容 |
|---|---|
| 対象ユーザー | 食品EC運用担当者・EC事業責任者（中小企業の食品ECショップ）。月商50万〜300万円。BtoB卸取引を併用する製造直販・D2C事業者 |
| 課題 | 賞味期限管理が「在庫数」だけでは制御できず、出荷可否判断が担当者の目視・記憶に完全依存している |
| 現状の問題 | EC上に賞味期限情報がないためFEFO（期限近い順の先入先出）が実現できず、残日数基準以下の商品出荷を防ぐ仕組みがない |
| 影響 | 担当者属人化による運用崩壊リスク |

| 項目 | 内容 |
|---|---|
| ユーザータイプ | 食品EC運用担当者（1名兼務が多い）・EC事業責任者 |
| 利用シーン | 商品入荷時のロット登録、日次の在庫状況確認、出荷前の賞味期限バリデーション |
| 頻度 | 入荷時・日次・受注時（自動）|
| 規模 | 月商50万〜300万円の中小企業の食品ECショップ。EC専任なし〜1〜3名 |

## 5. 提供価値（Value Proposition）

- **UX向上：** 出荷前の目視ロット確認作業をゼロへ。倉庫担当者の「出荷前日のロット手動当て直し」業務を半自動化。商品情報一括アップロードCSVで入荷時の登録負荷を最小化する。
- **作業効率改善：** 賞味期限ロット手作業確認・CSV加工時間を現行比50%以上削減。推定375時間/年/社の業務削減を実現する。
- **売上向上：** 期限切迫品を別商品（訳あり品・アウトレット品）として手動登録し割引販売することで、廃棄予定在庫を売上に転換。管理者が主体的に操作する設計でブランドリスクを回避。
- **リスク低減：** 出荷最低残日数の強制バリデーションで担当者属人化運用を排除し、業務継続性を担保。

---

## 6. 機能概要

在庫をロット（同一条件で製造・入荷された商品のひとまとまり）単位で管理し、賞味期限・製造日を属性として保持する。受注時にはFEFO（期限の近い順に先出し）で自動引当し、設定した残日数基準を下回るロットは出荷対象から自動除外する。また期限切迫ロットが発生した際は、管理者が任意のタイミングで別商品（訳あり品・アウトレット品）として手動登録し、割引販売機会を確保できる。

---

## 7. ユーザーストーリー

```
As a 食品EC運用担当者（1名兼務）
I want 受注時に賞味期限の近いロットから自動で在庫を引き当て、残日数が足りないロットは出荷不可にしたい
So that 出荷前の在庫ロット手作業確認をゼロにし、期限切れ商品の出荷トラブルをなくせる
```

## 8. 主要機能

| 機能 | 説明 |
|---|---|
| ① ロット登録 | 商品バリエーション単位に賞味期限・製造日・入荷日・ロット在庫数を登録。手動入力と商品情報一括アップロードCSVの両対応（CSVはMVPに含む）。※「ロット」とは同じ条件で製造・入荷された商品のひとまとまりのこと。同じ商品でも製造日が異なれば賞味期限が異なるため、ロット単位で在庫を管理することで「期限が近いものから出荷する（FEFO）」制御が初めて可能になる。 |
| ② FEFO自動引当 | 注文受付時に賞味期限が近い順のロットから自動引当。複数ロットにまたがる場合もトランザクション管理で整合性を担保。※「FEFO（First Expired, First Out）」とは「期限が近いものから先に出荷する」方式のこと。賞味期限が古いロットを優先して出荷することで、期限切れ廃棄を最小化する。一般的な倉庫管理のFIFO（先入れ先出し）の食品特化版にあたる。 |
| ③ 残日数バリデーション | 商品ごとに出荷可能な最低残日数（例：残8日以上）を設定。基準を下回るロットを引当対象から自動除外し、基準外商品の出荷を防止 |
| ④ アラートメール | 設定した残日数閾値を下回ったロットが発生した場合に管理者へ自動送信（日次バッチ）。既存在庫アラート機能を流用・拡張 |
| ⑤ 期限切迫品の別商品登録 | アラート対象となった期限切迫ロットを、管理者が任意のタイミングで別商品（例：訳あり品・アウトレット品）として新規登録できる機能。在庫を手動で切り替え、割引価格での販売機会を確保する。自動切替は行わず管理者が能動的に操作する設計とすることでブランドリスクを回避 |

## 15. メモ / 議論ログ

**推奨判断：Go — MVP開発推進**

**Go判断の根拠：**
- ヒアリング5社すべてで課題として挙がっており、現場ペインの実在は確認済み
- Shopify→makeshop乗り換えの「決定打」になりうる差別化機能
- MVPスコープを絞れば開発リスクは許容範囲

**MVP完了目標：2026年7月末**

**条件付きGo：**
- MVPスコープを「賞味期限・ロット属性管理＋FEFO引当＋出荷不可閾値設定」に絞る
- 食品安全に関する免責事項・利用規約の法務確認を先行して行う
- Tech Lead主導でFEFO排他制御の設計を先にスパイク（技術検証）する
- 食品ECを戦略的重点セグメントとして明示的に定義し、プロダクト戦略文書を更新する（Executive承認必要）

**着手前の必須アクション：**
1. Tech Lead主導でFEFO引当の技術スパイクを実施し、開発難易度・工数の再見積もりを行う（〜2週間）
2. 法務部門と「食品安全責任・免責事項」の利用規約文言を確認する（〜2週間）
3. 温度帯担当との合同DBスキーマ設計会議を設定する（〜2週間）
4. 加盟店へ「残日数計算基準」と「CSVフォーマット」の追加ヒアリングを実施する（〜1ヶ月）
5. 営業・CSデータから、賞味期限管理機能を必要とする食品EC加盟店候補数を定量化する（〜1ヶ月）

**Devil's Advocateからの主要指摘（要継続監視）：**
- 対象顧客（食品EC中小）の市場規模は思ったより小さい可能性あり → ヒアリング追加（10〜20社）で定量確認が必要
- 「食品EC特化」はプロダクト戦略の「特殊な業界専用機能は優先度低」原則と矛盾 → Executiveがプロダクト戦略に「食品EC戦略セグメント」を明示的に追加することで解消
